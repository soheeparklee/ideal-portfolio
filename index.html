Trouble Shooting - Instance connection lost error

  • Situation

    After 5 to 6 hours after deployment with nohup, the instance connection was lost.
    However, Instance on AWS EC2 was running without problem.

  • Tried and Achieved

    • Rebooted AWS EC2 instance • Rechecked nohup.out file. • SWAP • Updated for spring cloud in build.gradle file • Configured Hakari

  • Result

    The cause of the first error was that the degraded version of Spring Boot parent and Spring cloud version was not compatible. → solved by updating the build.gradle file • OOM error → allocated 4G of SWAP to prevent memory over commitment • The configuration of HikariCP was too long, connections were staying in the pool for longer than they should → solved by configuring HakariCP MaxLifetime to 30 minutes

Trouble Shooting - Too many connections error

  • Situation

    • After connecting the frontend and backend, there was a 500 error.
    When several users were trying to access the database, there was a Too many connections error.

  • Tried and Achieved

    • Updated settings CORS error • Checked MariaDB settings • Calculated rates • Too little_number of connection pools: default is 30 • Too long wait_timeout: default is 28800ms(8h) • Reboot MySql server

  • Result

    Caculated the rates of Cache Miss Rate(2.40%), Connection Miss Rate(17%), and Connection usage(53%). → Shortened wait_timeout from 28800ms to 180ms. → Increased max_connections from 30 to 300 for Connection usage. • The error “Too many connections” no longer appears on server.log. • The database works without problems even with many users updating at the same time. • As more connections were being made, the default options were no longer effective. • It is important for the application to be flexible and scalable.
